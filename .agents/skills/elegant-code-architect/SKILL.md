# Elegant Code Architect Skill

## 触发条件

1. 用户要求进行“大功能重构”、“新增大模块”或要求“规范、优雅地写代码”时触发。
2. 作为一个高级智能体，你应该在面对复杂的业务需求时**主动**调用此原则，而不是像初级工程师一样直接开始堆砌代码。

## Instructions (执行步骤)

当你接手一个具有一定复杂度的开发任务时，请严格按照以下“提案”和“执行”两个阶段进行：

### 阶段一：架构设计与方案提案（强制拦截）

1. **分析与设计**：
   - 不要立即修改任何代码文件！
   - 根据工业界最主流、先进且优雅的编程范式（如中间件模式、职责链模式、单向数据流、Vite 插件钩子最佳实践、MCP 通信规范等）来思考。
   - 考虑代码的**可扩展性（Extensibility）**和**规范性（Maintainability）**，为未来可能的变更留出接口或抽象层。

2. **向用户出具《技术设计方案》**：
   向用户输出一份结构清晰的方案，必须包含以下内容：
   - **设计理念**：采用什么主流模式或思想（例如：为什么用单例 Context 而不是散装的全局变量，为什么要隔离客户端与服务端的逻辑等）。
   - **涉及目录/文件**：列出将要新增或修改的核心文件。
   - **扩展性考量**：简述这段代码以后如果需要扩展新功能，为什么会很方便。
   - **关键伪代码/接口定义**：展示最核心的 Type / Interface 或骨架，供用户审查。

3. **强制挂起等待**：
   - 方案输出完毕后，**必须停止操作**，并询问用户：“您是否同意这个设计方案？如果没问题，我就开始编写并覆盖代码了。”
   - **未经用户明确同意，绝对禁止提前写入或修改文件。**

### 阶段二：规范编码阶段（获批后执行）

1. 用户回复“同意”、“开始吧”后，开始修改文件。
2. 实际输出代码时，必须 100% 遵守刚才约定的设计方案。
3. 严格遵循项目中已有的 `project_context.md` 里的所有规则（如绝对禁止 any、颜色规范等）。
4. 完成后向用户汇报修改结果，并可建议下一步的测试或 PR 流程。
