# Create Standard PR Skill

## 触发条件

1. **严格被动触发**：本技能**禁止**智能体在完成局部的开发、重构或调试子任务后自动、主动地触发运行。
2. 只有当用户发出明确的指令（例如“帮我提交代码”、“保存当前工作”、“开始提 PR”或“合并代码”）时，智能体才能触发此流程。
3. **日常开发中**，智能体修改完代码文件后，应保持修改状态在本地工作区，等待用户进一步的联合修改或开发指令，严禁“改一个文件就提交一次”。

## Instructions (执行步骤)

当你需要为用户创建 PR 时（且仅在收到用户明确指示后），请严格按照以下“计划确认阶段”和“执行阶段”两部分执行：

### 阶段一：计划确认阶段（强制拦截）

1. **状态检查与计划拟定**：
   - 运行 `git status` 和 `git diff` 查看当前所有更改。
   - **防突击与单一功能校验**：评估改动范围。如果发现当前改动包含多个不相关的改动（或者改动文件极多），必须将它们拆分成多个独立、细粒度的 PR 分步计划（每个 PR 仅做一件事）。
2. **向用户阐明任务并等待授权**：
   - 向用户详细汇报你的提交计划：准备分几个 PR 提交？每个 PR 包含哪些文件？拟定的 PR 标题是什么？
   - **强制挂起**：汇报完毕后，必须**停止当前动作，等待用户明确确认**（例如回复“开始吧”或“让你做才开始做”）。**在未经用户同意前，绝对禁止执行任何实质性的代码添加、提交或 PR 创建操作。**

### 阶段二：执行阶段（获批后按计划执行）

3. **分支安全校验 (No Direct Main Push)**：
   - 获取当前分支名。如果当前处于 `main` 或 `master` 分支，**必须禁止直接提交和推送**。
   - 引导或协助用户切换到遵循命名规范的新分支（格式：`类型/简短描述`，例如 `feat/voice-controls`）。
2. **代码质量与多模块构建验证 (主分支可运行保障)**：
   - 运行 `npm run build` 测试打包（通过 `tsup`），确保无致命的 TypeScript 编译错误。
   - 保障合并后主分支代码随时可运行。如有报错，中止提交流程并提醒用户。
3. **依赖检查与版权声明评估**：
   - 检查 `package.json` 是否有改动。若有新依赖（如 `@babel/code-frame`），记录并在 PR 描述中说明。
   - 询问或自行分析是否复用了旧代码片段，准备在 PR 中注明来源。
4. **生成 Commit 并提交**：
   - 根据确认的计划生成符合 Conventional Commits 规范的 message。
   - 执行 `git add <目标文件>` 和 `git commit -m "<你的message>"`。
5. **推送到远程**：
   - 执行 `git push origin <当前分支名>`。
6. **创建 Pull Request (严格的描述与一致性校验)**：
   - **二次审阅**：确保你要写的 PR 描述与实际改动严格一致。
   - **起草 PR 描述**：自动草拟符合规范的 PR 描述，必须包含核心段落（PR 标题、功能描述、实现思路、测试方式），以及如适用时的（第三方库依赖声明、代码复用声明、团队分工）。
   - 将描述写入本地临时文件 `.agents/temp_pr_body.md`，然后运行 `gh pr create`，成功后删除临时文件。
7. **合并与本地同步**：
   - 运行 `gh pr merge <PR号> --merge` 自动将 PR 合并入主分支。
   - 合并完成后，在本地执行 `git checkout main` 并运行 `git pull origin main` 同步，保持工作区干净。
